CRM Backend · Manager Role

Manager & Team Management API

Complete reference for all Team and HR endpoints. A manager is any user whose RBAC role carries the team.write permission. Managers can own and operate multiple teams simultaneously.

Base /api Auth Bearer <token> Teams · Members · Requests · HR

Manager Role Concept

What is a "Manager"?

There is no separate Manager database model. A manager is simply a CRM User whose assigned RBAC role carries the team.write permission. This gives them full authority to create teams, add/remove members, approve requests, and track attendance.

A single manager can oversee multiple teams at the same time. Each team can have its own Team Lead — a member promoted to the LEAD role within that team — who handles day-to-day requests that then flow back to the manager for approval.

Hierarchy

  ┌──────────────────────────────────────────────┐
  │               MANAGER (User)                 │
  │         role with  team.write perm           │
  │                                              │
  │   ┌───────────────┐   ┌───────────────┐      │
  │   │    Team A     │   │    Team B     │  ...  │
  │   │               │   │               │      │
  │   │  [LEAD] user  │   │  [LEAD] user  │      │
  │   │  [MEMBER] ... │   │  [MEMBER] ... │      │
  │   │  [ASSISTANT]  │   │  [ASSISTANT]  │      │
  │   └───────────────┘   └───────────────┘      │
  └──────────────────────────────────────────────┘

  TeamRequest flow (initiated by LEAD):
    LEAD creates request  →  status: "pending"
    MANAGER approves/rejects  →  status: "approved" | "rejected"
    On approval: TeamMember record created/deleted/transferred

Multi-Team

A user can be a member of multiple teams simultaneously via the TeamMember join table.

Dual-Path Writes

team.write holders act directly. Team LEADs create TeamRequest records that require manager approval.

Audit Trail

Every create / update / delete / assign operation writes to the AuditLog table automatically.

POST /api/teams/:teamId/members team.write (or LEAD)

Assigns one or more users to a team. Provide either userId (single) or userIds (array). Already-existing members are automatically skipped.

Request Body

{
  "userId": "clusr4...",           // optional — single user
  "userIds": ["clusr5...", "clusr6..."], // optional — multiple users
  "role": "MEMBER"                 // optional — "MEMBER" | "ASSISTANT_LEAD" | "LEAD" (default: MEMBER)
  // At least one of userId or userIds is required
}

Response 200 — Direct assignment (manager)

{
  "assigned": [

  ],
  "requests": []

Response 200 — LEAD-initiated (pending approval)

{
  "assigned": [],
  "skipped": [],
  "requests": [
    {
      "id": "clreq1...",
      "type": "assign",
      "requesterId": "clusr1...",
      "targetUserId": "clusr4...",
      "toTeamId": "clteam1...",
      "status": "pending",
      "createdAt": "2025-06-01T12:00:00.000Z"

  ]

Manager approves or rejects a pending team request. On approval the corresponding database action is performed automatically.

Request TypeAction Performed

removeDeletes TeamMember from fromTeamId
transferTransaction: deletes old TeamMember, creates new TeamMember in toTeamId

HR — Location Requests

Base path: /api/hr/location-requests

Privacy model: Only the original requester (manager/lead) can read the location result. The target user must actively respond with their coordinates — the app cannot push location silently.
POST /api/hr/location-requests location.request

Manager sends a location request to a team member. The member is notified and must respond within the TTL window.

Request Body

{
  "targetUserId": "clusr4...",    // required
  "expiresInSeconds": 300,        // optional — minimum 5 seconds (default 300)
  "note": "Client visit check"    // optional
}

Response 201

{
  "requestId": "clloc1...",
  "expiresAt": "2025-06-01T12:05:00.000Z"
}
POST /api/hr/location-requests/:id/respond requireAuth (target user only)

Target user responds with their current coordinates. Only the user the request was sent to can respond.

Request Body

{
  "latitude": 28.6139,                         // required
  "longitude": 77.2090,                        // required
  "accuracy": 10.5,                            // optional — metres
  "recordedAt": "2025-06-01T12:03:00.000Z"    // optional — defaults to now
}

Response 200

{ "ok": true }
GET /api/hr/location-requests/:id/result location.read (requester only)

Reads the coordinates submitted by the target user. Only the original requester can access this endpoint.

Response 200 (responded)

{
  "coords": {
    "latitude": 28.6139,
    "longitude": 77.2090,
    "accuracy": 10.5,
    "recordedAt": "2025-06-01T12:03:00.000Z"
  }
}

Response 404 (not yet responded or expired)

{ "status": "not_found" }

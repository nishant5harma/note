Lead Webhook Intake

This document explains how external systems (portals, Google Forms, websites, etc.) should call POST /api/leads/webhook to create and update leads inside the CRM.

Endpoint /api/leads/webhook Auth HMAC signature Creates leads · Links duplicates · Queues assignment

What this endpoint does

/api/leads/webhook is the only entry point for external sources to send new leads into the CRM. It is used by:

  • Property portals (99acres, MagicBricks, etc.)
  • Google Forms (lead capture forms)
  • Website contact / enquiry forms
  • Any custom integration that wants to push lead data

For every call, the backend will:

  1. Verify the HMAC signature (security).
  2. Normalize & extract fields like name, email, phone, source, externalId.
  3. Compute a dedupe hash based on phone/email/source/externalId.
  4. Either link to an existing lead or create a new lead.
  5. Create a LeadWebhookEvent row for full audit history.
  6. Insert an initial LeadAssignment and queue the lead for automatic assignment.

Security & HMAC Signature

The webhook is not protected by JWT tokens. Instead, each integration uses a shared secret key and HMAC (hash-based message authentication code).

How it works:

  1. The sender (Google Apps Script, portal backend, website backend, etc.) has a secret key (e.g. WEBHOOK_SECRET_DEFAULT or a provider-specific secret like WEBHOOK_SECRET_GOOGLE_FORMS).
  2. Before sending, it serializes the JSON body to a string bodyJson.
  3. Computes HMAC_SHA256(bodyJson, secret) and encodes as hex.
  4. Sends that value in a header such as X-Signature.
  5. The CRM backend recomputes the same HMAC and rejects the request if signatures do not match.

This ensures only trusted systems (that know the secret) can create or update leads.

Endpoint Details

Base path: /api/leads/webhook

POST /api/leads/webhook HMAC only · no JWT

Headers

Header Required Description
Content-Type Yes Use application/json.
x-lead-source / x-source Recommended Source identifier: magicbricks, 99acres, google_forms, website, etc.
x-idempotency-key Optional Unique key per submission to de-duplicate retries.
x-signature Yes HMAC SHA256 signature of the raw JSON body, hex-encoded. The backend recomputes and verifies this.

Recommended Payload Shape

The backend can work with arbitrary JSON, but this is the canonical shape we recommend for all sources (portals, forms, website, etc.): 

{
  "provider": "google_forms",          // string - which system is sending the lead
  "externalId": "GF-2025-06-01-001",   // string - unique ID from that system
  "name": "Ravi Kumar",                // string - full name
  "email": "ravi@example.com",         // string - email address (optional but recommended)
  "phone": "+919876543210",            // string - phone number in international format
  "source": "google_forms",            // string - marketing/source tag used for reporting
  "payload": {                         // object - full original payload from the source
    /* everything else from the form / portal */
  }
}

Important: at least one of phone or email should be provided for meaningful deduplication.

Responses

1) New lead created · 201

{
  "ok": true,
  "note": "created",
  "leadId": "cllead1...",
  "webhookEventId": "clwh1..."
}

2) Existing lead linked · 200

{
  "ok": true,
  "note": "linked",
  "leadId": "clleadExisting...",
  "webhookEventId": "clwh2..."
}

3) In-flight / duplicate · 200

{
  "ok": true,
  "note": "processing in-flight",
  "eventId": "clwh3..."
}

4) Signature invalid · 401/403

{
  "error": "invalid signature"
}

Example: Google Forms Integration

You can connect a Google Form directly to the webhook using a small Apps Script that runs on every form submission.

  1. Open your Google Form → Script editor.
  2. Paste the script below (adjust field names and secret).
  3. Create a trigger for onFormSubmit on Form submit.
// Apps Script example
const WEBHOOK_URL = 'https://your-domain.com/api/leads/webhook';
const SHARED_SECRET = 'YOUR_HMAC_SECRET';

function onFormSubmit(e) {
  const resp = e.namedValues; // map: question title -> [answers]

  const body = {
    provider: 'google_forms',
    externalId: resp['Timestamp'][0],          // or any unique field
    name: resp['Name'][0],
    email: resp['Email'][0],
    phone: resp['Phone'][0],
    source: 'google_forms',
    payload: resp
  };

  const bodyJson = JSON.stringify(body);
  const signature = Utilities.computeHmacSha256Signature(bodyJson, SHARED_SECRET);
  const sigHex = signature
    .map(function (b) { return ('0' + (b & 0xff).toString(16)).slice(-2); })
    .join('');

  UrlFetchApp.fetch(WEBHOOK_URL, {
    method: 'post',
    contentType: 'application/json',
    payload: bodyJson,
    headers: {
      'x-lead-source': 'google_forms',
      'x-idempotency-key': resp['Timestamp'][0],
      'x-signature': sigHex
    },
    muteHttpExceptions: true
  });
}

Example: Website Backend Integration

For a website (Next.js, Node, Laravel, etc.), you should call the webhook from your backend — never directly from the browser — and sign the payload using your shared secret. 

// Node.js / Next.js API route example
import crypto from 'crypto';
import fetch from 'node-fetch';

const WEBHOOK_URL = 'https://your-domain.com/api/leads/webhook';
const SECRET = process.env.WEBHOOK_SECRET_DEFAULT;

export async function sendLeadToWebhook(lead) {
  const body = {
    provider: 'website',
    externalId: lead.id,       // stable unique ID from your system
    name: lead.name,
    email: lead.email,
    phone: lead.phone,
    source: 'website',
    payload: lead
  };

  const bodyJson = JSON.stringify(body);

  const signature = crypto
    .createHmac('sha256', SECRET)
    .update(bodyJson, 'utf8')
    .digest('hex');

  const res = await fetch(WEBHOOK_URL, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-lead-source': 'website',
      'x-idempotency-key': lead.id,
      'x-signature': signature
    },
    body: bodyJson
  });

  const data = await res.json();
  return { status: res.status, data };
}

Frontend Integration Guide (How to Receive Leads)

High-level flow

User fills frontend form (website / app / Google Form)
        ⬇
Your frontend sends data to your own backend (e.g. /api/contact)
        ⬇
Your backend builds the canonical lead JSON + HMAC
        ⬇
CRM endpoint POST /api/leads/webhook
        ⬇
Lead is created/linked and appears in CRM Leads UI

Important: the frontend never calls /api/leads/webhook directly. Only trusted backends (which know the HMAC secret) are allowed to talk to the webhook.

Step 1 — Configure CRM webhook URL & secret

  • In the CRM backend .env, set one of these secrets: 
    WEBHOOK_SECRET_DEFAULT=some-long-random-string
WEBHOOK_SECRET_GOOGLE_FORMS=google-forms-secret-if-used
WEBHOOK_SECRET_FACEBOOK=facebook-secret-from-app
  • Expose the backend over HTTPS so the webhook is reachable, for example:
    • Dev: http://<your-dev-ip>:4000/api/leads/webhook
    • Prod: https://your-domain.com/api/leads/webhook
  • Use the same secret value in all senders (website backend, Apps Script, Zapier, etc.).
  • The secret used is selected by x-lead-source:
    • If x-lead-source: google_forms then backend uses WEBHOOK_SECRET_GOOGLE_FORMS (or falls back to default).
    • If provider is missing, backend uses WEBHOOK_SECRET_DEFAULT.

Step 2 — Frontend form submits to your own API

Example React/Next.js form that posts to /api/contact on your website backend: 

// React example
async function handleSubmit(e) {
  e.preventDefault();

  const body = {
    name: e.target.name.value,
    email: e.target.email.value,
    phone: e.target.phone.value,
    message: e.target.message.value,
  };

  const res = await fetch('/api/contact', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  });

  const data = await res.json();
  console.log('CRM webhook response:', data);
}

Step 3 — Your backend forwards to /api/leads/webhook

In your website backend route (for example /api/contact), convert the form data to the canonical lead payload and forward it to the CRM: 

// /api/contact backend handler (Node/Next.js/Express)
import crypto from 'crypto';
import fetch from 'node-fetch';

const CRM_WEBHOOK_URL = 'https://your-domain.com/api/leads/webhook';
const SECRET = process.env.WEBHOOK_SECRET_DEFAULT;

export async function handleContact(req, res) {
  const { name, email, phone, message } = req.body;

  const payload = {
    provider: 'website',
    externalId: `web-${Date.now()}`, // unique per submission
    name,
    email,
    phone,
    source: 'website',
    payload: { name, email, phone, message },
  };

  const bodyJson = JSON.stringify(payload);

  const signature = crypto
    .createHmac('sha256', SECRET)
    .update(bodyJson, 'utf8')
    .digest('hex');

  const crmRes = await fetch(CRM_WEBHOOK_URL, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-lead-source': 'website',
      'x-idempotency-key': payload.externalId,
      'x-signature': signature,
    },
    body: bodyJson,
  });

  const data = await crmRes.json();
  res.status(crmRes.status).json(data);
}

After this is in place, every time the frontend form is submitted, a lead will be created (or linked) in the CRM and show up in the Leads section with source = "website".

Step 4 — Testing from frontend

  1. Open your frontend form, fill in test data, submit.
  2. Check your website backend logs to confirm it hit /api/contact.
  3. Check the CRM logs / Leads UI to see that a new lead was created with the correct name, phone, email, and source.
  4. Once working, repeat the same pattern for other forms (e.g. landing pages, mobile app forms).

Step 5 — How to view all leads (API + Postman)

After webhook integration is active, you can view all leads using these protected endpoints. These require JWT auth and lead.read permission.

1) Login to get token

POST /api/auth/login
Content-Type: application/json

{
  "email": "superadmin@example.com",
  "password": "SuperSecurePassword123!"
}

2) List leads

GET /api/leads?page=1&limit=50&q=ravi&source=google_forms&status=NEW
Authorization: Bearer <accessToken>

Filters supported: q, source, status, stage, priority, assignedToId, assignedTeamId, from, to.

3) Get lead details

GET /api/leads/:id
Authorization: Bearer <accessToken>

This returns the full lead with assignments, webhook events, notes, follow-ups, escalations, and stage history.

Frontend: View all leads (list + detail)

To show leads in your frontend, use the new endpoints GET /api/leads and GET /api/leads/:id. These are protected endpoints and require: JWT token + lead.read permission.

1) Login (get accessToken)

POST /api/auth/login
Content-Type: application/json

{ "email": "superadmin@example.com", "password": "SuperSecurePassword123!" }

2) Fetch leads list

// Example (React / Web)
const API_BASE_URL = "http://<YOUR_IP>:4000";

async function fetchLeads(token) {
  const params = new URLSearchParams({
    page: "1",
    limit: "50",
    // optional filters:
    // source: "google_forms",
    // status: "NEW",
    // q: "ravi",

  const res = await fetch(`${API_BASE_URL}/api/leads?${params.toString()}`, {
    headers: { Authorization: `Bearer ${token}` },
  });
  return await res.json(); // { ok, items, meta }
}

3) Fetch one lead (detail page)

async function fetchLeadById(token, leadId) {
  const res = await fetch(`${API_BASE_URL}/api/leads/${leadId}`, {
    headers: { Authorization: `Bearer ${token}` },
  });
  return await res.json(); // { ok, data }
}

Tip: After a webhook creates a lead, your UI can refresh the list by calling GET /api/leads.

Deduplication & Lead Creation

Deduplication

  • The backend computes a dedupe hash from: phone, email, source, and externalId.
  • It first tries to find an existing lead by externalId.
  • If not found, it tries by dedupe hash.
  • If still not found, a new lead is created.

Lead Creation

  • A new Lead record is inserted with normalized fields (name, phone, email, source, etc.).
  • An initial LeadAssignment is created with:
    • assignedTo = null
    • assignedBy = "system"
    • method = "auto"
    • attempt = 0
  • The lead ID is attached back to the LeadWebhookEvent created for this webhook call.

Webhook Events & Audit

  • Every request creates a LeadWebhookEvent row (even if duplicate), which stores:
    • Raw payload
    • Provider / source
    • Signature metadata
    • Associated leadId if matched/created
  • When a new lead is created, an audit record is also written: action = "lead.create", resource = "lead", including source and webhookEventId.

Automatic Assignment

  • After storing the event and creating the lead, the system enqueues a job in the assignmentQueue.
  • Background workers pick up this job and:
    • Find the correct team via SourcePool configuration (if any).
    • Use round-robin + capacity rules to pick an agent.
    • Assign the lead to that agent and update UserCapacity.
    • Send socket & push notifications to the assigned user.
  • If automatic assignment cannot find any suitable agent (e.g., all at capacity), the lead can eventually be escalated and appear in /api/leads/assignment/admin/escalated.

Lead Flow (Webhook → Round Robin → Follow-ups)

A frontend-facing explanation of how leads enter the CRM, get assigned automatically via round-robin (with capacity checks), and then move through follow-ups and status updates.

Intake POST /api/leads/webhook View GET /api/leads Follow-ups /api/leads/followup

End-to-end flow

1) External source sends a lead
   - Website form / Google Form / Facebook Lead Ads / Portal
   - Calls POST /api/leads/webhook (HMAC secured)

2) Intake service normalizes + dedupes
   - Computes dedupeHash using phone/email/source/externalId
   - Creates LeadWebhookEvent for every request
   - Creates Lead if not found (or links if found)

3) Assignment job is enqueued
   - Creates a LeadAssignment row (method: "auto", assignedTo: null)
   - Background workers process the assignment queue

4) Worker assigns to an employee (agent)
   - Determines team via SourcePool mapping (optional)
   - Chooses next eligible member by round-robin
   - Enforces capacity (UserCapacity.used < maxOpen)
   - Updates Lead.assignedToId (+ assignedAt)

5) If no agent available → escalation
   - Lead status can become UNASSIGNED_ESCALATED
   - Manager can manually assign

6) Agent actions
   - Create follow-ups, update status/stage/priority, view stage history

Who receives leads automatically?

Leads are assigned automatically to employees/agents in the round-robin pool. Managers will only get leads automatically if they are also in the team member pool (i.e. part of the assignment roster).

  • Employees/Agents: get assigned based on team membership + capacity.
  • Managers: typically monitor escalations and manually assign when required.
  • Everyone: never “broadcast assigned”. Each lead has exactly one assignee at a time (assignedToId).

Setup guide (Teams, Pools, Employees)

In this CRM, the “pool” for round-robin is simply a Team. The pool members are rows in TeamMember. Source-based routing is controlled by SourcePool.

You need 4 things for perfect lead receiving

  1. Team(s) created (example: Website Team, Facebook Team).
  2. Employees added as team members (these are the round-robin candidates).
  3. SourcePool mapping: lead source → which team should receive it.
  4. Capacity set (default maxOpen=10) so routing respects workload.

Step 1 — Create a Team (Manager)

Endpoint (JWT + team.write):

POST /api/teams
Authorization: Bearer <token>
Content-Type: application/json

{ "name": "Website Team", "leadId": null }

Create separate teams if you want different sources to go to different employees.

Step 2 — Add employees to the Team (Pool members)

Endpoint (JWT + team.write):

POST /api/teams/<teamId>/members/bulk
Authorization: Bearer <token>
Content-Type: application/json

{
  "userIds": ["<empUserId1>", "<empUserId2>", "<empUserId3>"],
  "role": "MEMBER"
}

These users become the round-robin pool. Users can be in multiple teams.

Step 3 — Decide the source values (very important)

When calling POST /api/leads/webhook you send a body field source. This value is used for routing. 

// Examples (choose and keep consistent)
source = "website"
source = "google_forms"
source = "facebook"
source = "google_ads"
source = "99acres"

Step 4 — Create SourcePool mapping (source → team)

Important

There is currently no HTTP endpoint in this backend to create/edit SourcePool. SourcePool is a database table, so you configure it via Prisma/SQL/seed.

Prisma example (run in a script/seed or Prisma Studio):

await prisma.sourcePool.upsert({
  where: { source: "website" },
  update: { teamId: "clteamWebsite...", active: true },
  create: { source: "website", teamId: "clteamWebsite...", strategy: "round_robin", active: true },
});

Repeat for google_forms, facebook, etc.

Step 5 — Capacity tuning (optional but recommended)

Capacity is stored in UserCapacity. If a user hits maxOpen, round-robin skips them. 

await prisma.userCapacity.upsert({
  where: { userId: "<empUserId>" },
  update: { maxOpen: 25 },
  create: { userId: "<empUserId>", maxOpen: 25, used: 0 },
});

Intake endpoint (creates/links leads)

HMAC secured. No JWT.

POST /api/leads/webhook HMAC

Required headers:

Content-Type: application/json
x-lead-source: google_forms | website | facebook | ...
x-signature: sha256=<hmac_hex_of_body>
x-idempotency-key: <optional>

Canonical body shape:

{
  "provider": "google_forms",
  "externalId": "GF-001",
  "name": "Ravi Kumar",
  "email": "ravi@example.com",
  "phone": "+919876543210",
  "source": "google_forms",
  "payload": { /* full raw source payload */ }
}

Responses:

201 { "ok": true, "note": "created", "leadId": "...", "webhookEventId": "..." }
200 { "ok": true, "note": "linked",  "leadId": "...", "webhookEventId": "..." }

Deduplication rules

  • Backend computes dedupeHash from phone, email, source, externalId.
  • Lookup order: first by externalId, then by dedupeHash.
  • Every request creates a LeadWebhookEvent (audit trail), even if it links to an existing lead.

Source → Team routing (SourcePool)

If you want different sources to go to different teams, configure a SourcePool record: source → teamId.

Example: source = google_forms routes to Team A; source = facebook routes to Team B.

Round-robin assignment logic

Assignment uses a Redis pointer per team (or global), which is incremented to select the next member. Team members are considered in join-order and rotated. 

Inputs:
  - lead.source (optional, for team routing)
  - team members (TeamMember rows)
  - redis pointer (lead:rr:team:<teamId>)

Output:
  - chosen userId (agent)

Pseudo:
  members = teamMembers ordered by joinedAt asc
  idx = pointer % members.length
  candidate = members[idx]
  pointer++

Capacity (UserCapacity)

Each agent has a capacity row: UserCapacity with maxOpen and used. Auto-assignment only claims a user if used < maxOpen.

  • If candidate is at capacity, worker tries the next round-robin candidate.
  • When a lead reaches terminal status (WON/LOST/UNQUALIFIED), capacity is released (used--).

Escalation

If no eligible agent can be found after retries/attempts, the lead is escalated and becomes visible to managers for manual intervention.

GET /api/leads/assignment/admin/escalated JWT + lead.escalation 
GET /api/leads/assignment/admin/escalated?page=1&limit=50
Authorization: Bearer <token>

Manual assignment (manager override)

Managers can assign any lead to a specific user.

POST /api/leads/assignment/:id JWT + lead.assign 
POST /api/leads/assignment/<leadId>
Authorization: Bearer <token>

{ "userId": "<assigneeUserId>" }

Frontend API: View leads

These endpoints power the Leads list + Lead detail screens in frontend.

New: Frontend/Admin can manage Source → Team mapping

  • New sources (e.g. facebook, google_ads, website) should be mapped to a team via API.
  • This avoids Prisma/seed changes every time a new lead source is added.
  • Protected by: JWT + lead.assign permission.

Frontend “who is this lead assigned to?”

  • GET /api/leads gives assignedToId.
  • Use GET /api/users (or /api/users/:id) to map assignedToId → user.name.
  • Lead detail page can call GET /api/leads/:id to show full assignment history (assignments[]).

Testing checklist (does lead receiving work?)

  1. Webhook call succeeds:
    • Send POST /api/leads/webhook with correct HMAC headers.
    • Expect 201 created or 200 linked.
  2. Lead appears in list:
    • Login and call GET /api/leads with JWT token.
    • Confirm the lead shows with the correct source, name, phone.
  3. Assignment happens:
    • Wait for worker to assign, then re-fetch lead detail GET /api/leads/:id.
    • Confirm assignedToId is set and capacity used increased for that user.
  4. Follow-up workflow works:
    • Create follow-up: POST /api/leads/followup/:id/followups
    • List follow-ups: GET /api/leads/followup/:id/followups
  5. Terminal status releases capacity:
    • Set status to WON or LOST using PATCH /api/leads/followup/:id/status.
    • Confirm capacity used decreases for the assignee.

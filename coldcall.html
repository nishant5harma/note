Cold Call Module

End-to-end flow + all endpoints for uploading cold call data, distributing to teams, agent calling workflow (pull/lock/attempt/complete), retries & auto-unlock, reports, analytics, and quotas.

Base path: /api/coldcall

Auth: All ColdCall endpoints are behind JWT (because AppRouter mounts it as /api/coldcall with requireAuth).

RBAC: each route also checks requirePermission("coldcall", "...") (examples below).

Flow overview (how it works)

ColdCall assignment = 2 steps

  • Team assignment happens at distribute time (assignedTeamId).
  • Employee assignment happens at pull time via lock (lockUserId + lockExpiresAt).
  1. Upload file (Excel/CSV) → backend creates a ColdCallBatch + many ColdCallEntry rows. Duplicate control is via dedupePolicy (keep/skip).
  2. Distribute batch to teams → batch entries get assignedTeamId (routing logic in distribute service). There is also a preview endpoint for dry-run.
  3. Agent calling loop: agent hits pull to claim next entry. Backend uses Postgres row-locking (FOR UPDATE SKIP LOCKED) so two agents never get the same entry.
  4. Lock + TTL: claimed entries move to in_progress and get lockUserId + lockExpiresAt. Agent can refresh lock or release it.
  5. Attempts: agent can log multiple attempts per entry. This does not finalize the entry.
  6. Complete: finalize entry as done with a response/disposition/summary. Optionally it can create a CRM Lead (source = cold_call) and an optional lead follow-up.
  7. Retry: after completion, backend evaluates retry rules and can schedule the entry to return to pending later (BullMQ).
  8. Auto-unlock: a repeating BullMQ job clears expired locks so stuck entries come back to pending.
  9. Reports/Analytics: report endpoints summarize batches/entries/teams/agents. analytics endpoints provide team/agent performance + leaderboard.

Upload

Important about teamIds on upload

  • If you pass mode=manual + teamIds, entries are pre-filled with assignedTeamId = teamIds[0].
  • Otherwise, entries will be assigned to teams later by distribute.

Supported file formats (important)

  • Supported: .xlsx, .xls, .csv
  • Not supported currently: .xml

Backend currently parses XLSX/CSV (XML import is not implemented).

Expected columns (header names)

Importer tries common aliases. Minimum recommended: phone OR email. Name is optional.

  • Phone: phone, mobile, contact, phone number
  • Email: email, mail
  • Name: name, fullname, full name
  • Any extra columns: saved into payload JSON per entry

Phone is normalized to digits-only before saving.

Bulk Lead import (from ColdCall)

  • ColdCall upload creates ColdCallEntry, not Leads directly.
  • Lead creation happens on agent action: POST /api/coldcall/pull/entries/:id/complete with leadConversion.createLead=true.
  • Created lead will have source=cold_call and is linked back to the entry.

Batch distribution

How distribution decides the team

  • Which entries are considered? default = only entries with assignedTeamId = null. Use force=true to re-distribute all.
  • Which teams are available? taken from batch.teamConfig.selectedTeamIds, else derived from batch.routingConfig.rules.
  • Per-entry routing rules (optional): location, budget. If a rule returns a teamId, that team is used.
  • Fallback: if no rule decides, backend uses Redis weighted round-robin: rrKey = coldcall:rr:batch:<batchId>, sequence expanded from weights.

Example routingConfig (location + budget + fallback RR)

{
  "rules": [
    {
      "kind": "location",
      "config": {
        "map": {
          "mumbai": "TEAM_MUMBAI_ID",
          "pune": { "teamId": "TEAM_PUNE_ID" }
        }
      }
    },
    {
      "kind": "budget",
      "config": {
        "ranges": [
          { "min": 0, "max": 5000000, "teamId": "TEAM_AFFORDABLE_ID" },
          { "min": 5000001, "max": 20000000, "teamId": "TEAM_PREMIUM_ID" }
        ]
      }
    },
    { "kind": "default", "config": {} }
  ]
}

If rules do not match, RR will pick from available teams (optionally weighted by batch.teamConfig.weights).

Agent workflow (pull / lock / attempt / complete)

How “next call” is chosen

  • Backend picks only status=pending and teams in your membership.
  • Ordering: priority DESC, then createdAt ASC.
  • Concurrency-safe: Postgres FOR UPDATE SKIP LOCKED.

Who got which entry (team vs employee)

  • Team = ColdCallEntry.assignedTeamId (after distribute).
  • Current agent holding it = lockUserId (only while status=in_progress).
  • Attempts history = ColdCallAttempt rows created via /attempt and final attempt created at /complete.
  • To view in UI, use GET /api/coldcall/report/entries with filters (batchId/teamId/userId/status/response).

RBAC: Give ColdCall access to employees

ColdCall access is permission-based (not superadmin-only)

  • Anyone can upload/distribute/call if their role contains the required permission keys.
  • Frontend should provide an Admin screen to create roles and assign permissions/users.

ColdCall permission keys (use these in UI)

  • coldcall.upload — upload Excel/CSV → create batches
  • coldcall.distribute — distribute batch entries to teams
  • coldcall.call — agent workflow (pull/attempt/complete)
  • coldcall.read — reports + analytics read
  • coldcall.assign — reassign entries
  • coldcall.admin — quota admin actions

RBAC endpoints (Admin Panel)

Base path: /api/rbac (all require JWT + role/permission permissions).

  • List permissions (for checkboxes): GET /api/rbac/permissions (needs permission.read)
  • Create/List roles: POST /api/rbac/roles, GET /api/rbac/roles (needs role.write/role.read)
  • Assign permission to role: POST /api/rbac/roles/:id/permissions (needs role.write)
  • Remove permission from role: DELETE /api/rbac/roles/:id/permissions/:permId (needs role.write)
  • Get role permissions: GET /api/rbac/roles/:id/permissions (needs role.read)
  • Assign role to a user: POST /api/rbac/roles/assign-to-user (needs role.write)
  • Remove role from a user: DELETE /api/rbac/roles/user/:userId/role/:roleId (needs role.write)

Common errors (and fixes)

1) “Missing permission: coldcall.call”

  • Meaning: user’s role does not include coldcall.call.
  • Fix: assign permission via RBAC endpoints above (role → permissions → user).

2) “no-team-membership” (pull returns 400)

  • Meaning: user is not a member of any team, so backend cannot choose entries for them.
  • Fix: add user to a team (TeamMember). After that, pull can work.

3) “no-available-entry”

  • Meaning: there are no pending entries assigned to your teams.
  • Fix: ensure batch is distributed to the same teams (assignedTeamId matches your teamIds), and entries are pending.

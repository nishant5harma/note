HR Module

Complete guide for HR module — Attendance, Location Tracking, Device Management, and Consent. All endpoints, permissions, and flows.

Base path: /api/hr

Auth: All HR endpoints require JWT (requireAuth) + per-route RBAC permission.

Mounted at: AppRouter.use("/hr", requireAuth, HrRouter)

4 Sub-modules

Attendance

Employee checkin/checkout + admin list. /api/hr/attendance

Permissions: attendance.self, attendance.read

Location Requests

Supervisor requests employee location on-demand (ephemeral, Redis-stored). /api/hr/location-requests

Permissions: location.request, location.read

Device Management

Register/unregister push notification devices. /api/hr/devices

Permissions: device.register, device.unregister, device.read

Consent

Create/revoke user consent for location tracking, photo, terms. /api/hr/consent

Permissions: consent.write, consent.read

Attendance Endpoints

POST /api/hr/attendance/checkin JWT + attendance.self 
POST /api/hr/attendance/checkin
Authorization: Bearer <token>
Content-Type: application/json

{
  "photoRef": "https://storage/selfie.jpg",  // optional
  "latitude": 19.076,
  "longitude": 72.877,
  "accuracy": 10,
  "locationTs": "2026-03-18T09:00:00.000Z",  // optional ISO string
  "note": "On time"
}

Only 1 checkin per day per user allowed. Creates Attendance, LocationLog, LocationAccessAudit + AuditLog in one transaction.

POST /api/hr/attendance/checkout JWT + attendance.self 
POST /api/hr/attendance/checkout
Authorization: Bearer <token>

Sets checkoutAt on today's attendance. Only 1 checkout per day. Must have checked in first.

GET /api/hr/attendance JWT + attendance.read 
GET /api/hr/attendance?userId=<id>&teamId=<id>&from=2026-03-01&to=2026-03-31&page=1&limit=100
Authorization: Bearer <token>

Admin/Manager can filter by user, team, date range. Returns rows with user + team info.

Location Request Endpoints

Flow

  1. Supervisor creates request → employee device gets real-time push + socket event.
  2. Employee app responds with GPS coordinates.
  3. Coordinates stored ephemerally in Redis (TTL ~60s).
  4. Supervisor reads result once (one-time read, then deleted from Redis).
POST /api/hr/location-requests JWT + location.request 
POST /api/hr/location-requests
Authorization: Bearer <token>
Content-Type: application/json

{
  "targetUserId": "<employeeId>",
  "expiresInSeconds": 60,
  "note": "Client visit verification"
}
POST /api/hr/location-requests/:id/respond JWT (target user only) 
POST /api/hr/location-requests/<requestId>/respond
Authorization: Bearer <token>
Content-Type: application/json

{
  "latitude": 19.076,
  "longitude": 72.877,
  "accuracy": 8,
  "recordedAt": "2026-03-18T10:05:00.000Z"
}
GET /api/hr/location-requests/:id/result JWT + location.read (requester only) 
GET /api/hr/location-requests/<requestId>/result
Authorization: Bearer <token>

Returns { coords: { latitude, longitude, accuracy, recordedAt } }. One-time read (deleted from Redis after).

Real-time — Socket.IO

The backend runs a Socket.IO server alongside the HTTP server. It is used for real-time bidirectional communication — primarily for location requests, push-style notifications, and future presence tracking. When a user is offline, the system falls back to FCM (Firebase) or WebPush (VAPID) via registered devices.

Architecture Overview

Mobile App / Browser
      │
      │  1. HTTP handshake: Authorization: Bearer <JWT>  (or auth.token)
      ▼
Socket.IO Server (authSocketMiddleware)
      │  verifies JWT → loads user + permissions from DB
      │  attaches socket.user, socket.data.userId, socket.permissions
      ▼
On "connect" event:
  socket.on("register", { userId }) → socket.join(userId)
                                     → userSocketMap.set(userId, socketIds)
      │
      │  Server can now do: io.to(userId).emit("event", payload)
      ▼
All devices logged in as the same user receive the event simultaneously

1 — Connect & Authenticate

Pass the JWT in the Socket.IO handshake. Two ways are supported:

// Option A — via auth object (recommended for mobile/React Native)
const socket = io("https://api.yourcrm.com", {
  auth: { token: "eyJhbGci..." }   // JWT access token
});

// Option B — via Authorization header (web browsers)
const socket = io("https://api.yourcrm.com", {
  extraHeaders: { Authorization: "Bearer eyJhbGci..." }
});

The server middleware (authSocketMiddleware) reads socket.handshake.auth.token or Authorization header, verifies the JWT, loads the user and their permissions, then stores them on the socket: socket.data.userId, socket.permissions.

2 — Register (Join User Room)

After connecting, the client must emit a register event so the server can route messages to this user's room. Without this step no targeted events will be delivered.

socket.on("connect", () => {
  // userId is ignored if JWT auth was used (server uses auth identity)
  socket.emit("register", { userId: currentUser.id }, (ack) => {
    console.log(ack); // { status: "ok", userId: "..." }
  });
});
Security note: If JWT auth was provided, the server ignores the userId in the payload and uses the identity from the token. The client-supplied userId is only used as a fallback when no auth token is present (unauthenticated sockets).

3 — Events to Listen (Server → Client)

Event Name Who receives it When fired Payload shape
location.fetch_request Target employee Supervisor calls POST /api/hr/location-requests { type, requestId, silent, ts }
location.responded Requesting supervisor Employee responds via POST /api/hr/location-requests/:id/respond { requestId, coords: { latitude, longitude, accuracy, recordedAt }, responderId }
LOCATION_FETCH Target employee (push fallback) Same as above — also sent as FCM/WebPush data payload { type: "LOCATION_FETCH", requestId, silent: true, ts }
LOCATION_RESPONSE Requester supervisor (push fallback) Employee responds — also sent as FCM/WebPush to supervisor { type: "LOCATION_RESPONSE", requestId, coords, responderId }
push Any targeted user Generic push via pushToUser() internally Arbitrary { type, ...payload }

Client-side listener examples:

// Employee: listen for supervisor location requests
socket.on("location.fetch_request", async (data) => {
  // { type: "LOCATION_FETCH", requestId, silent, ts }
  const coords = await getCurrentGPS();  // get device GPS
  await api.post(`/api/hr/location-requests/${data.requestId}/respond`, {
    latitude: coords.lat,
    longitude: coords.lng,
    accuracy: coords.accuracy,
    recordedAt: new Date().toISOString(),
  });
});

// Supervisor: listen for employee response
socket.on("location.responded", (data) => {
  // { requestId, coords: { latitude, longitude, accuracy, recordedAt }, responderId }
  showLocationOnMap(data.coords);
});

// Generic push notification handler
socket.on("push", (data) => {
  showInAppNotification(data);
});

5 — Push Notification Fallback (Offline Users)

When a user is offline (not connected to Socket.IO), the server uses registered push devices to deliver messages via FCM or WebPush. The device must be registered using POST /api/hr/devices/register.

Delivery priority order:

  1. Socket.IO emit — instant, if user is connected (userSocketMap checked)
  2. FCM (Firebase Cloud Messaging) — Android & iOS via firebase-admin
  3. WebPush (VAPID) — Web browsers via web-push

Environment Variables needed for push:

# Firebase (Android/iOS)
GOOGLE_APPLICATION_CREDENTIALS_JSON={"type":"service_account",...}
# or
GOOGLE_APPLICATION_CREDENTIALS_PATH=/path/to/firebase-key.json

# WebPush (Web browsers)
VAPID_PUBLIC_KEY=BNxxx...
VAPID_PRIVATE_KEY=xxx...
VAPID_SUBJECT=mailto:admin@yourcrm.com

# Optional TTLs
PUSH_DEFAULT_TTL_SECONDS=60
LOCATION_REQUEST_TTL_SECONDS=60
LOCATION_REQUEST_EXPIRY_SECONDS=60

FCM message shape (data-only, silent):

{
  token: "<fcm-device-token>",
  data: {
    type: "LOCATION_FETCH",     // all values stringified
    requestId: "<id>",
    silent: "true",
    ts: "2026-03-18T10:00:00Z"
  },
  android: { priority: "high" },
  apns: { payload: { aps: { "content-available": 1 } } }   // silent push iOS
}

7 — Multi-Device Support (userSocketMap)

One user can be logged in on multiple devices simultaneously. The server maintains a userSocketMap: Map<userId, Set<socketId>>. When a message is emitted to io.to(userId), Socket.IO delivers it to all connected sockets in that room (all devices).

// Server behavior
userSocketMap.get(userId) → Set { "socket_A", "socket_B" }   // 2 devices online
io.to(userId).emit("location.fetch_request", payload);        // both devices receive it

// On disconnect, the socketId is removed
// If set becomes empty → userId deleted from map → user is "offline"

The emitToUser() function checks the map first. If the set is empty (user offline), it returns false — which triggers the push notification fallback.

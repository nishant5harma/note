Lead Management API

Complete reference for all Lead endpoints — intake via webhooks, automatic & manual assignment, escalations, follow-ups, status, stage history, and analytics (funnel & ratings).

Base /api/leads Auth Bearer <token> Webhooks · Assignment · Follow-ups · Analytics

Lead Lifecycle Overview

High-level Flow

1) Intake
   • External portals / forms call /api/leads/webhook.
   • Payload is normalized into a Lead and WebhookEvent.
   • A pending LeadAssignment is created and queued.

2) Automatic Assignment
   • Background workers pick up the assignment job.
   • Strategy: round-robin + capacity limits + source pools (team-based).
   • Lead is assigned to an agent; their capacity is reserved.

3) Manual Overrides & Escalation
   • Managers can manually re-assign a lead at any time.
   • When attempts/timeouts exceed thresholds, the lead is escalated.
   • Escalated leads are visible in `/assignment/admin/escalated`.

4) Follow-ups & Status
   • Agents add follow-ups with due dates, notes, dispositions, ratings.
   • Status updates (e.g. WON / LOST / UNQUALIFIED) create audit follow-ups.
   • Capacity is released when a lead reaches a terminal status.

5) Analytics
   • Admins can inspect all follow-ups, funnel by stage, and rating summaries.

Permissions

Permission Scope Used By
lead.read Read lead data, follow-ups, stage history, analytics Agents, team leads, managers
lead.write Create follow-ups, update status/stage/priority Agents, team leads
lead.assign Manual (re)assignment of leads Managers / supervisors
lead.escalation View escalated leads queue Managers / supervisors
lead.read (admin) Admin follow-up list, funnel, ratings Super admin / reporting roles

Lead Intake (Webhook)

Base path: /api/leads/webhook

POST /api/leads/webhook HMAC only (no JWT)

Receives raw lead payloads from external portals (99acres, MagicBricks, website forms, etc.). This endpoint is not authenticated with JWT — it is protected by an HMAC signature middleware instead.

Security

  • External system sends X-Hub-Signature / HMAC header (implementation-specific).
  • Backend verifies the signature over the raw body using a shared secret.
  • Requests failing verification are rejected before reaching business logic.

Headers

Header Required Description
x-lead-source / x-source Recommended Provider identifier (e.g. magicbricks, 99acres, website).
x-idempotency-key Optional Unique key to dedupe retry attempts for the same event.
HMAC header(s) Required Provider-specific header(s) for signature verification.

Request Body

Accepts arbitrary JSON (or text) body. Under the hood, it is normalized into a lead shape: 

{
  "provider": "magicbricks",
  "externalId": "MB-1234567890",
  "name": "Ravi Kumar",
  "email": "ravi@example.com",
  "phone": "+919876543210",
  "source": "magicbricks",
  "payload": {
    /* full raw provider payload */
  }
}

Responses

Depending on dedupe outcome and existing lead state:

New lead created · 201

{
  "ok": true,
  "note": "created",
  "leadId": "cllead1...",
  "webhookEventId": "clwh1..."
}

Existing lead linked · 200

{
  "ok": true,
  "note": "linked",
  "leadId": "clleadExisting...",
  "webhookEventId": "clwh2..."
}

In-flight / Duplicate · 200

{
  "ok": true,
  "note": "processing in-flight",
  "eventId": "clwh3..."
}

Business Rules

  • Deduplication uses a hash of phone, email, source, externalId.
  • A Redis guard ensures each dedupe key is processed once at a time.
  • Existing leads are looked up by externalId first, then by dedupe hash.
  • Every webhook creates a LeadWebhookEvent row for full auditability.
  • When a new lead is created, an initial LeadAssignment is inserted and queued for automatic assignment.

Manual Assignment & Escalation

Base path: /api/leads/assignment

POST /api/leads/assignment/:id lead.assign

Manually assigns a lead to a specific user. Can be used to override automatic routing, handle escalations, or reassign leads between agents.

Path Parameters

id Lead ID (required)

Request Body

{
  "userId": "clusr123..."   // required — user to assign this lead to
}

Response 200

{
  "ok": true
}

Business Rules

  • Cancels any pending auto-assignment jobs tied to this lead (via Bull queues).
  • Updates Lead.assignedToId, assignedAt, and clears assignedTeamId.
  • If a previous LeadAssignment exists, it is updated with method "manual", assignedBy, and manualAssignedAt.
  • If no assignment history exists, a new LeadAssignment record is created.
  • Capacity is adjusted:
    • New assignee’s UserCapacity.used is incremented.
    • Previous assignee’s used is decremented (down to a minimum of 0).
GET /api/leads/assignment/admin/escalated lead.escalation

Lists escalated leads — leads which failed automatic assignment or exceeded follow-up thresholds and now require manual intervention by a manager.

Query Parameters

Param Type Description
page number Page number (default 1)
limit number Items per page (default 50, max 100)

Response 200

{
  "ok": true,
  "data": [
    {
      "id": "clleadEscalated1...",
      "name": "Ravi Kumar",
      "phone": "+919876543210",
      "email": "ravi@example.com",
      "status": "UNASSIGNED_ESCALATED",
      "createdAt": "2025-06-01T09:00:00.000Z",
      "assignedTeamId": "clteam1..."  // may be null
    }
  ],
  "meta": { "page": 1, "limit": 50, "total": 3 }
}
POST /api/leads/assignment/admin/escalated/:id/assign lead.assign

Manually assigns an escalated lead to a specific agent. Internally, this delegates to the same manual assignment service used by /api/leads/assignment/:id.

Path & Body

// Path
POST /api/leads/assignment/admin/escalated/:id/assign

// Body
{
  "userId": "clusr123..."  // required — user to assign this escalated lead to
}

Response 200

{
  "ok": true
}

Follow-ups & Lead Status

Base path: /api/leads/followup

POST /api/leads/followup/:id/followups lead.write

Creates a follow-up entry for a lead — typically a call, visit, or note scheduled by an agent.

Request Body

{
  "assignedTo": "clusr123...",              // optional — user responsible for this follow-up
  "dueAt": "2025-06-02T10:00:00.000Z",      // optional — ISO date
  "note": "Call the client after site visit", // optional, max 2000 chars
  "disposition": "Interested, needs approval", // optional, max 500 chars
  "rating": 4,                              // optional, integer 0–5
  "status": "pending"                       // optional, "pending" | "done" | "skipped"
}

Response 200

{
  "ok": true,
  "data": {
    "id": "clfup1...",
    "leadId": "cllead1...",
    "assignedTo": "clusr123...",
    "dueAt": "2025-06-02T10:00:00.000Z",
    "note": "Call the client after site visit",
    "disposition": "Interested, needs approval",
    "rating": 4,
    "status": "pending",
    "createdAt": "2025-06-01T12:00:00.000Z"
  }
}

Notifications

  • If assignedTo is set, a socket event followup.created and a push notification FOLLOWUP_CREATED are sent to that user.
GET /api/leads/followup/:id/followups lead.read

Lists follow-up entries for a single lead, most recent first, with pagination.

Query Parameters

Param Type Description
page number Page number (default 1)
limit number Items per page (default 50, max 200)

Response 200

{
  "ok": true,
  "items": [ /* LeadFollowUp[] ordered by createdAt desc */ ],
  "meta": { "page": 1, "limit": 50, "total": 12 }
}
PATCH /api/leads/followup/:id/status lead.write

Updates the status, stage, and/or priority of a lead. Also creates a follow-up note capturing the reason, rating, and previous values, and optionally records stage history.

Request Body

{
  "status": "WON",                       // required — LeadStatus enum
  "stage": "CLOSURE",                    // optional — LeadStage enum
  "priority": "HOT",                     // optional — LeadPriority enum
  "disposition": "Booked a 3BHK unit",   // optional — max 500 chars
  "note": "Deal closed after 2nd visit", // optional — max 2000 chars
  "rating": 5                            // optional — integer 0–5
}

Response 200

{
  "ok": true,
  "data": {
    "lead": { /* updated Lead object */ },
    "followUp": {
      /* follow-up record describing this status change, including meta.previousStatus/stage/priority */
    }
  }
}

Business Rules

  • A distributed lock is used to prevent concurrent status updates for the same lead.
  • When moving into a terminal status (WON, LOST, UNQUALIFIED) from a non-terminal state:
    • The assignee’s UserCapacity.used is decremented (down to 0), freeing up capacity.
  • Changes to stage are recorded into LeadStageHistory with fromStage, toStage, changedBy, and note.
  • Notifications are emitted to the current assignee:
    • Socket event lead.status.changed
    • Push notification LEAD_STATUS_CHANGED

Returns the full stage-change history for a given lead, ordered from oldest to newest.

Admin Follow-ups & Analytics

Base path: /api/leads/followup/admin

GET /api/leads/followup/admin lead.read

Admin view of follow-ups across the entire system, with filters and pagination.

Query Parameters

Param Type Description
disposition string Filter by disposition (exact match or implementation-defined)
assignedTo string Filter by assignee user ID
from ISO string Only follow-ups with createdAt >= from
to ISO string Only follow-ups with createdAt <= to
page number Page number (default 1)
limit number Items per page (default 100, max 500)

Response 200

{
  "ok": true,
  "items": [ /* LeadFollowUp[] matching filters */ ],
  "meta": { "page": 1, "limit": 100, "total": 245 }
}
GET /api/leads/followup/admin/funnel lead.read

Returns a funnel breakdown of active leads by stage, optionally scoped to a team or user.

Query Parameters

Param Type Description
teamId string Optional. Restrict to leads assigned to this team.
userId string Optional. Restrict to leads assigned to this user.

Response 200

{
  "ok": true,
  "data": [
    { "stage": "NEW",        "count": 120 },
    { "stage": "CONTACTED",  "count": 80 },
    { "stage": "CLOSURE",    "count": 25 },
    { "stage": "WON",        "count": 10 }
  ]
}
GET /api/leads/followup/admin/ratings lead.read

Returns aggregate ratings for leads, grouped by stage. Useful for measuring quality/conversion perception at different stages of the funnel.

Query Parameters

Param Type Description
groupBy string Currently supported: stage. Other values will return an empty array.

Response 200

{
  "ok": true,
  "data": [
    { "stage": "NEW",       "avgRating": 3.4, "cnt": 25 },
    { "stage": "CONTACTED", "avgRating": 4.1, "cnt": 40 },
    { "stage": "WON",       "avgRating": 4.8, "cnt": 10 }
  ]
}

Enums & Statuses

LeadStatus (examples)

NEW Freshly created, no contact made yet
CONTACTED Agent has spoken or interacted with lead
IN_PROGRESS Active follow-ups, not yet decided
WON Converted / booked — terminal (capacity released)
LOST Lost / not proceeding — terminal (capacity released)
UNQUALIFIED Not a fit / spam — terminal (capacity released)
UNASSIGNED_ESCALATED Escalated for manual assignment

LeadStage (examples)

NEW Just captured from source
CONTACTED First contact made
SITE_VISIT Site visit scheduled/done
CLOSURE Negotiation / closing in progress

LeadPriority (examples)

HOT High-intent lead, needs urgent follow-up
WARM Engaged but needs nurturing
COLD Low-intent, long-term nurture

Follow-up Status

pending Upcoming / yet to be completed
done Follow-up completed
skipped Skipped or no longer needed
